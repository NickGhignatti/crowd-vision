Sensor Data Communication

To ensure high throughput and low latency, the system utilizes a unidirectional data tunnel orchestrated via Redis Pub/Sub. The architecture strictly separates concerns: production, filtering, and distribution are handled by three independent services that never call each other directly.

Core Components

  1. Sensor Service (The Producer) — Blindly emits the absolute truth of the physical world.

    • Publishes full telemetry payloads to telemetry:raw.
    • Each message includes a type field identifying the metric ("temperature", "peopleCount", "airQuality").

  2. Contracts Service (The Filter) — A Rust/Axum microservice that acts as the intelligent gatekeeper.

    • Subscribes to telemetry:raw.
    • Maintains an O(1) in-memory DashMap mapping each Building ID to its list of allowed metric types.
    • For each incoming message, looks up which buildings have subscribed to that metric type and republishes the payload to their dedicated channel.
    • Publishes the tailored payload to telemetry:filtered:<building_id>.

  3. Socket Service (The Distributor) — Stateless WebSocket relay.

    • When a client connects via WebSocket, it subscribes to telemetry:filtered:<building_id>.
    • Pushes data directly down the TCP socket to the browser.

Why Type-Based Filtering?

The raw telemetry payload is a flat JSON object with a type discriminator field:

{ "type": "temperature", "buildingId": "alpha", "roomId": "r1", "timestamp": 1718000000, "value": 22.4 }

The Contracts Service checks whether payload["type"] is in the building’s allowed_columns list. If it is, the entire payload is forwarded unchanged. This approach:

  • Preserves context: Downstream consumers always receive roomId, timestamp, and buildingId — they never need to re-fetch spatial context.
  • Keeps the filter O(1): A Vec::contains on a short list of metric keys is faster than any field-by-field JSON manipulation.
  • Scales to N buildings: Because fan-out is performed by iterating the DashMap, a single telemetry:raw message can be simultaneously forwarded to multiple buildings that share the same metric subscription — with no duplicate Redis subscriptions.

Frontend Subscription Pattern

The Vue client listens for telemetry on a single "telemetry" Socket.IO event (not per-channel). The socket-service emits this event only to the building:<id> room, so each client only receives data for buildings it has joined.

Lifecycle rules enforced in useSensorData:

  1. On buildingId change: emit('subscribe_building', id) to join the Socket.IO room.
  2. Register handler via socket.on('telemetry', telemetryHandler) — always capturing the exact function reference.
  3. On cleanup: socket.off('telemetry', telemetryHandler) — must match the same reference to prevent listener accumulation.
  4. Data mutations are batched to the next requestAnimationFrame via triggerRef(), decoupling Vue’s scheduler from the Three.js render loop. This is critical for the digital twin view’s 60fps camera animation.
