Sensor Data Communication

To ensure high throughput and low latency, the system utilizes a unidirectional data tunnel orchestrated via Redis Pub/Sub. The architecture strictly separates concerns: production, routing, and distribution are handled by three independent services that never call each other directly.

Core Components

  1. Sensor Service (The Producer) — Blindly emits the absolute truth of the physical world.

    • Publishes full telemetry payloads to telemetry:raw.
    • Each message includes a type field identifying the metric ("temperature", "peopleCount", "airQuality").

  2. Contracts Service (The Router) — A Rust/Axum microservice that routes each reading to the building it belongs to.

    • Subscribes to telemetry:raw.
    • Maintains an O(1) in-memory DashMap of known buildings (and their dashboard column preferences, used by the HTTP API).
    • For each message it reads the event’s own buildingId and republishes the payload to that one building’s channel — a single O(1) lookup, never a fan-out across buildings.
    • Publishes the payload to telemetry:filtered:<building_id>.

  3. Socket Service (The Distributor) — Stateless WebSocket relay.

    • When a client connects via WebSocket, it subscribes to telemetry:filtered:<building_id>.
    • Pushes data directly down the TCP socket to the browser.

Why Per-Building Routing?

The raw telemetry payload is a flat JSON object carrying its own buildingId:

{ "type": "temperature", "buildingId": "alpha", "roomId": "r1", "timestamp": 1718000000, "value": 22.4 }

The Contracts Service reads payload["buildingId"] and, if that building is known, forwards the entire payload unchanged to telemetry:filtered:<buildingId>. This approach:

  • Routes to exactly one building: keying on the event’s own buildingId means a reading reaches only that building’s feed — no cross-building leak, no iterating every preference.
  • Preserves context: Downstream consumers always receive roomId, timestamp, and buildingId — they never need to re-fetch spatial context.
  • Stays O(1): a single DashMap lookup per event, regardless of how many buildings exist.

No metric-type gate

Telemetry is not filtered by the dashboard’s column selection. Every metric a building produces is forwarded; which columns the dashboard table shows is decided client-side from the same preferences. This keeps other consumers — notably the 3D model view, which always needs temperature and air-quality — live regardless of the table’s column setup.

Frontend Subscription Pattern

The Vue client listens for telemetry on a single "telemetry" Socket.IO event (not per-channel). The socket-service emits this event only to the building:<id> room, so each client only receives data for buildings it has joined.

Subscription and live data are owned by a shared Pinia store (stores/sensorData.ts), consumed through the useBuildingSensor(buildingId, type) composable. Many components can want the same building’s metric (e.g. the 3D model and the room list both read temperature); they share one reference-counted bucket instead of each opening its own fetch + subscription + handler.

Lifecycle, centralised in the store:

  1. The first acquire of a (type, building) creates a bucket, REST-fetches the latest snapshot, and — if it’s the building’s first metric — emits subscribe_building to join the room.
  2. A single global socket.on('telemetry', …) handler dispatches each event to its bucket via an O(1) (type, buildingId) map lookup — there are no per-component handlers to accumulate.
  3. The last release of a bucket tears it down; once a building has no buckets left, the store emits unsubscribe_building.
  4. Data mutations are batched to the next requestAnimationFrame via triggerRef() on a shallowRef, decoupling Vue’s scheduler from the Three.js render loop — critical for the digital twin’s 60fps camera animation.
  5. On socket reconnect the store re-emits subscribe_building for every active building and refetches, recovering anything missed while disconnected.
