Kubernetes Configuration

Kubernetes is the intended environment for staging and production, providing scaling, self-healing, and stateful data management. This page is the operational guide — how to deploy, inspect, and update the cluster. For the cluster’s internal design (StatefulSets, ingress rewriting, init containers), see Deployment & Kubernetes.

All Kubernetes operations run through just recipes from the repository root, so you do not need to memorise long kubectl invocations. The recipes assume a running cluster (k3d, k3s, Minikube, or a managed provider) with your kubectl context set correctly.

First-time setup and the core lifecycle

# First time on a new cluster — installs the ingress controller, creates the
# namespace, injects secrets from .env, then applies all manifests:
just k8s setup

# Re-apply all manifests after a change (idempotent — safe to run repeatedly):
just k8s apply

# Tear everything down (deletes the namespace and ALL data inside it):
just k8s delete

k8s-delete is destructive

just k8s delete removes the namespace and every PersistentVolumeClaim in it — all database data is permanently lost, with no undo. To pause without losing data, use just k8s stop and later just k8s start.

Deployment order

Manifests must be applied in dependency order, because a service crashes if its database is not yet reachable. just k8s apply applies them in exactly this sequence; the manual equivalent is shown for understanding.

Secrets

Secrets are created directly on the cluster from your local .env and are never committed. just k8s secrets reads values such as JWT_SECRET, VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, the agent provider keys, GHCR_USERNAME, and GHCR_TOKEN, and creates the corresponding Secret objects. It is idempotent — safe to re-run after rotating a key.

Namespace before secrets

The namespace must exist before secrets can be created. Run kubectl apply -f k8s/namespace.yml (or the full just k8s setup) before just k8s secrets if invoking it in isolation.

Agent provider key naming

The current agent-service-secret generator forwards the legacy GOOGLE_API_KEY and DEEPSEEK_API_KEY names, while the recommended provider setup uses OPENROUTER_API_KEY. Before deploying the agent, confirm the generated secret contains a key compatible with its configured LLM_BASE_URL. Kubernetes also does not deploy the documentation ingester or Langfuse — see the Agent Service guide.

Day-to-day operations

just k8s status                 # pods, deployments, services, StatefulSets
just k8s logs auth-service      # tail logs (works for Deployments and StatefulSets)
just k8s restart auth-service   # rolling restart one service (picks up a new :latest)
just k8s restart-all            # restart every deployment (e.g. after rotating secrets)
just k8s rollout auth-service   # watch a rolling update to completion
just k8s validate               # dry-run every manifest without touching the cluster

A rolling restart starts new pods and waits for their readiness probe to pass before terminating the old ones, so a healthy service never goes fully down. If a rollout misbehaves, roll back instantly:

kubectl rollout undo deployment/auth-service -n crowdvision

Image management

Production images (GHCR)

just k8s build        # build all images, tagged for GHCR
just k8s push         # push to GHCR (requires: docker login ghcr.io)
just k8s build-push   # build and push in one step

Local k3d (no registry)

Build images locally and import them straight into the k3d cluster, skipping GHCR entirely:

just k8s build   # build all images locally
just k8s load    # import them into k3d
just k8s apply   # apply manifests

Image pull policy for local images

When using locally loaded images, set imagePullPolicy: IfNotPresent in the deployment manifests. With Always, Kubernetes attempts a GHCR pull and ignores the locally imported image.

Local DNS

For local k3d access, add the ingress hostname to your machine’s hosts file:

just k8s hosts
# Prints: 127.0.0.1   crowdvision.local

Edit the hosts file as administrator (Windows: C:\Windows\System32\drivers\etc\hosts; Linux/macOS: /etc/hosts) and add that line.
