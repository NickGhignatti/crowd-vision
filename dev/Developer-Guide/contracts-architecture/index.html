Contracts Service Architecture

Bounded context: Telemetry Distribution · Stack: Rust / Axum · Code-level walkthrough: Contracts Service

The contracts-service realises the Telemetry Distribution context. It sits between the raw reading stream and the browser feed, shaping each building’s live view. It is the one service written in Rust, chosen for throughput on a hot path that inspects every reading.

Architectural Style

Not a layered REST service. It runs two concurrent components in one process: an HTTP API for managing preferences, and a background stream processor that filters telemetry. Its frequently-read state is kept in memory, with a database behind it for durability.

Internal Structure

Key Architectural Decisions

  • Rust on the hot path. The filter inspects a high-frequency stream; the language choice is an architectural decision to keep that path cheap and predictable.
  • In-memory projection. Preferences are read on every reading but changed rarely, so they are held in a concurrent in-memory structure for lock-free reads; the database is touched only on startup (hydration) and on change (cold path).
  • Filter by preference, fan out per building. A reading is forwarded only to the buildings whose preference includes its metric; one reading can fan out to many building feeds.
  • Gather, do not define, the catalog. The service discovers each upstream service’s self-describing metric catalog and aggregates it, so new metrics appear without any change here — an Open Host Service consuming each upstream’s Published Language.

Integration

  • Subscribes to the raw-reading channel and publishes per-building filtered feeds on the broker — see Communication & Data Flow.
  • Discovers metric catalogs from sensor-service and twin-service; receives default preferences seeded by twin-service.

For the concurrency model, the tunnel internals, persistence status, and the API, see the Contracts Service internals page.
