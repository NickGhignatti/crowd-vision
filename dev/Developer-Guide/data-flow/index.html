Communication & Data Flow

CrowdVision uses two communication styles, chosen deliberately per interaction:

  • Synchronous (request/response) over HTTP through the gateway — for user-driven actions and for the few service-to-service calls that need an immediate answer.
  • Asynchronous (publish/subscribe) over Redis — for the high-frequency telemetry and alert flows, where producers must not be coupled to, or blocked by, consumers.

This page maps both, and then walks the two end-to-end real-time paths.

Synchronous Communication

All external requests enter through the single gateway, which routes by URL prefix (see the routing table in the Overview. A handful of service-to-service calls also happen over HTTP on the internal network:

CallerCalleePurpose
twin-servicesensor-serviceOn building creation/update, push a threshold baseline for the building’s rooms.
twin-servicecontracts-serviceSeed a new building’s default display preferences.
sensor-servicenotification-serviceForward a temperature reading that may warrant an alert.
notification-servicetwin-serviceResolve which domains a building belongs to before selecting recipients.
contracts-servicesensor-service / twin-serviceDiscover each service’s self-describing metric catalog.

These calls are intentionally few. The design keeps the request path short and avoids synchronous chains on the hot path; anything high-frequency goes over the broker instead.

Identity travels with the request

There is no central session store. auth-service mints a signed identity token; every other service verifies it locally and reads the principal’s memberships, roles, and domains from it. No service calls auth-service to authorize a request.

Asynchronous Communication

Redis is the publish/subscribe backbone. Three channels carry the event flow:

ChannelPublisherSubscriberCarries
telemetry:rawsensor-servicecontracts-serviceEvery accepted reading, as a normalized event.
telemetry:filtered:{buildingId}contracts-servicesocket-serviceReadings a given building has opted to display.
notificationsnotification-servicesocket-serviceAlert events for delivery to the browser.

Publishers never know who (if anyone) is listening, and the broker absorbs bursts, so a slow or restarting consumer cannot back-pressure a sensor.

Path 1: Live Telemetry to the Browser

The defining flow: a reading becomes a live update on the dashboard, shaped per building.

contracts-service is the shaping stage: it consults each building’s display preferences and forwards a reading only to the buildings that asked for that metric. socket-service then delivers it to exactly the clients watching that building (see Socket Service.

Path 2: Threshold Alert to the User

When a reading crosses a configured bound, the alert path runs in parallel to the telemetry path.

The alert path is owned by notification-service: it suppresses repeats with a short cooldown, resolves recipients via twin-service, and fans out over two channels — an in-app event through the broker and socket, and a Web Push to offline devices.

Why Two Styles

ConcernStyleReason
User actions (login, CRUD)SynchronousThe caller needs an immediate, ordered result.
Provisioning between servicesSynchronous (best-effort)Simple to reason about; failures are tolerated and re-tried later.
Telemetry & alertsAsynchronousHigh volume, many consumers, and producers that must never block.
