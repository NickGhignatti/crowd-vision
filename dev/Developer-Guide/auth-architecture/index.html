Authentication Service Architecture

Bounded context: Identity & Access · Stack: Node.js / Express · Code-level walkthrough: Authentication Service

The auth-service realises the Identity & Access context. Architecturally it is the platform’s trust anchor: it authenticates principals and issues the signed identity token that every other service verifies on its own.

Architectural Style

A layered Express service (see Service Anatomy built around a stateless authorization model. There is no server-side session — all the authority a request needs is carried in a signed token — so any instance can serve any request and no other service has to call back here to authorize.

Internal Structure

The HTTP surface is split across several controllers by concern, with the heavier logic isolated in dedicated services.

Key Architectural Decisions

  • Stateless JWT issuance. Memberships are embedded in the token at sign-in, so authorization everywhere else is a local check rather than a call back to this service.
  • Controllers split by concern (authentication, domain, administration) to keep each cohesive and small.
  • Specialised domain services. Token handling and the TOTP invite-code logic are kept out of the controllers so they can be reasoned about and tested in isolation.
  • Federation as account-linking. External SSO augments an existing account rather than creating one — a deliberate limit on what federation is allowed to do.
  • Cross-cutting concerns in middleware. Authorization and error shaping are factored into middleware instead of being repeated per controller (see Auth Service: Middleware.

Integration

  • Owns auth-db; no other service reads it.
  • Produces the identity token consumed platform-wide — see Communication & Data Flow.
  • Has no synchronous dependency on any other service.

For the login, SSO, and TOTP flows, the code-level decisions, and the API, see the Authentication Service internals page.
