Digital Twin Service

Implementation reference for the twin-service. For its architectural role, see Digital Twin Service Architecture; for the domain model, see Building Management.

A Node.js / Express service in TypeScript. It is the spatial source of truth: it stores buildings and rooms, and on every change it synchronises a lightweight clone to the dependent services.

Source Layout

PathResponsibility
src/router.tsRoute table for building CRUD, plus health, metrics, and the /contracts metric catalog.
src/controller/buildings.tsHTTP adapters over the building service.
src/services/buildings.tsCore CRUD logic and orchestration of downstream synchronisation.
src/services/names.tsPure helpers for normalising and backfilling building/room display names.
src/services/sensors.tsPushes the threshold clone to the sensor-service.
src/services/contractsService.tsSeeds default display preferences in the contracts-service.
src/models/building.tsMongoose schema for Building, Room, and the geometry value objects.

Data Model

interface IBuilding {
  id: string;        // server-generated UUID (crypto.randomUUID); unique
  name: string;
  rooms: Room[];     // embedded
  domains: string[]; // tenant names; indexed { domains: 1 }
}

interface Room {
  id: string;        // taken from the uploaded model file; stable
  name: string;
  capacity: number;
  position: { x: number; y: number; z: number };   // value object
  dimensions: { width: number; height: number; depth: number }; // value object
  color?: string;
}
  • id is generated server-side; a client-supplied id on creation is ignored.
  • The { domains: 1 } index makes Building.find({ domains }) an index seek rather than a collection scan.
  • Geometry is stored as _id: false subdocuments. The schema enables timestamps.

HTTP API

Method · PathDescription
POST /registerCreate a building from { name, rooms, domains }. Assigns a UUID, normalises names, persists, then synchronises downstream. Returns the full document. 201.
GET /building/:idFetch one building by UUID; backfills names on read. 404 if absent.
GET /buildings/:domainList buildings scoped to a tenant (domains contains :domain). Returns [] when none.
GET /domain/:buildingNameReturn the flattened list of domains for buildings matching the name. Used by notification-service for recipient resolution.
PATCH /building/:buildingIdUpdate name and/or domains (and an optional maxTemperature forwarded to the clone). Re-syncs.
PATCH /building/:buildingId/room/:roomIdUpdate a room’s name, color, or capacity. Re-syncs.
GET /contractsReturn this service’s metric catalog (roomName, roomMaxOccupancy).
GET /health/, GET /metrics/Liveness and Prometheus metrics.

Authentication is not enforced at this layer

As wired in router.ts, the building routes do not attach an authentication middleware. Tenant scoping is performed by the :domain path parameter against each building’s domains array, but the identity token is not verified inside the service. This is a gap relative to the intended design, where tenancy is enforced from the verified token. Treat token verification here as a required hardening step.

Name Normalisation

services/names.ts keeps display names consistent on both write and read.

  • normalizeBuildingName(name, id) resolves to the trimmed name, else id, else the literal "Building".
  • normalizeRoomNames(rooms) trims each room name, falling back to the room id.
  • backfillNames(building) re-applies these rules when a building is read and persists only if something changed — so older documents are healed lazily.

Downstream Synchronisation

A building’s structure is mirrored to two services. The two paths have different failure semantics, which matters operationally.

  • Sensor threshold clone (syncBuildingClone): a PUT carrying { name, rooms: [{ id, name }], maxTemperature? }. It is gated by shouldSyncThresholds() (skipped under test) and throws on a non-OK response, so a sensor-service failure fails the building operation.
  • Contracts preference seeding (initBuildingPreferences): a POST that is skipped under test and swallows errors (logged only), so it never blocks building creation.

Known gap: synchronous coupling

The sensor clone is an inline, synchronous REST call inside addBuilding/updateBuilding. The roadmap is to replace it with a published TWIN_UPDATED event so the sensor-service can update its copy asynchronously, decoupling the two services.

Error Handling

The service uses the same BaseError hierarchy and global error middleware as the auth-service (ValidationError 400, NotFoundError 404, …) — see Auth Service: Middleware. Building/room lookups raise NotFoundError when the target does not exist.
