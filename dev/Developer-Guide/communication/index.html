Communication Patterns in our Architecture

In a microservices architecture, how services communicate is just as important as what they do. CrowdVision’s long-term architectural goal is to restrict direct, synchronous service-to-service HTTP communication to prevent tight coupling and cascading failures.

However, we balance this strict decoupling with development pragmatism. Currently, the platform relies on four primary communication paradigms:

  1. API Gateway Routing (External to Internal)
  2. Event-Driven Pub/Sub (Internal Asynchronous)
  3. Stateless Trust (Cross-Service Authorization)
  4. Internal REST (Internal Synchronous - Transitional)

1. Client-to-Service (API Gateway)

All external communication from the Vue frontend or IoT sensors is routed through the Caddy Reverse Proxy.

The client never speaks to a microservice directly. Caddy acts as our ingress controller and API Gateway. It terminates the SSL/TLS connection, reads the URL path (e.g., /api/twin/buildings), strips the prefix, and forwards the request to the correct internal Docker/Kubernetes container on the isolated gateway-net.

2. Service-to-Service (Event-Driven via Redis)

When one microservice needs to inform another that a state change occurred (e.g., the Notification Service detects a crowded room and needs the Socket Service to update the UI), we prioritize asynchronous, event-driven architecture.

CrowdVision uses Redis as a central message broker implementing the Publish/Subscribe (Pub/Sub) pattern.

Why Redis Pub/Sub?

  • Decoupling: The Producer (notification-service) does not need to know the IP address, port, or even the existence of the Consumers. It simply fires the event into the ether and forgets about it.
  • Extensibility: If we want the Python-based agent-service to log an anomaly every time a room gets overcrowded, we don’t need to touch the Notification Service code. We simply instruct the Agent to SUBSCRIBE to the relevant Redis channel.

3. Stateless Trust (Cross-Service Authorization)

A major challenge in a Database-per-Service architecture is Authorization.

If a user asks the Twin Service to delete a building, the Twin Service needs to know if the user has Admin privileges. However, the Twin Service cannot query the Auth Database due to strict data isolation boundaries.

To solve this without making synchronous HTTP calls back to the Auth Service on every request, CrowdVision uses Stateless JSON Web Tokens (JWTs).

The Embedded Payload Pattern

When the Auth Service mints a token, it embeds the user’s IDomainMembership array directly into the JWT payload. Because the backend services share the same cryptographic JWT_SECRET, they can mathematically verify the token’s authenticity. Once verified, the receiving service simply reads the memberships from the payload to authorize the request—entirely eliminating network overhead.

4. Internal REST

⚠️ Architectural Note: This is currently a transitional pattern.

While Pub/Sub and Stateless JWTs handle events and authorization, there are instances in the current codebase where services must perform synchronous reads across bounded contexts (e.g., Service A immediately needs specific entity details owned by Service B to complete a transaction).

In these specific cases, services communicate directly via Internal HTTP REST calls.

  • These calls are made using internal Docker/Kubernetes DNS resolution (e.g., http://auth-service:3000).
  • They never traverse the public internet.
  • Future Roadmap: We are actively evaluating strategies to phase this out.

5. IoT Sensor to Service (Multiplexing)

Physical or simulated IoT sensors communicate their telemetry (temperature, people counts, etc.) to the backend using either standard HTTP POSTs.

The Multiplexing Pattern

Instead of spinning up a unique WebSocket server or connection for every physical room, CrowdVision uses a many-to-one multiplexing pattern:

  • Hundreds of sensor clients connect to the single socket-service instance via the Gateway.
  • To differentiate the continuous data streams, each sensor injects its corresponding roomId and sensorId directly into the JSON payload of every emitted event.
  • The backend aggregates, validates, and routes this telemetry to the sensor-service and connected frontends.
