Authentication Service

Implementation reference for the auth-service. For its architectural role and design rationale, see Authentication Service Architecture; for the domain model, see Identity & Access.

The service is a Node.js / Express application in TypeScript. It manages accounts, domains (tenants), role assignment, and federated sign-in, and it issues the JSON Web Token that the rest of the platform trusts.

Source Layout

PathResponsibility
src/index.tsExpress bootstrap: JSON body capture (with raw-body buffer for HMAC), cookie parsing, router mount, error and metrics middleware, Mongo connection.
src/router.tsRoute table; binds paths to controllers and attaches authentication/authorization middleware.
src/controller/HTTP adapters: authentication.ts, domain.ts, administration.ts, status.ts.
src/services/Business logic: authentication.ts, domain.ts, token.ts, totp.ts.
src/models/Mongoose schemas and shared types: account.ts, domain.ts, role.ts, token.ts, error.ts.
src/middlewares/authentication.ts (auth + HMAC guards), error.ts (global handler), metrics.ts (Prometheus).
src/config/config.ts (environment accessors), db.ts (Mongo), registry.ts (Prometheus registry).

Configuration

All runtime configuration is read through config/config.ts.

VariableDefaultPurpose
JWT_SECRET—HMAC secret used to sign and verify every token. Must be identical across all services.
VITE_SERVER_URLhttp://localhost:3000Public base URL of the gateway; used to build the OIDC redirect URI and to call sibling services.
CLIENT_URLhttp://localhost:5173Frontend origin; used as the post-SSO redirect target.
INTERNAL_ADMIN_SECRET—Shared secret for the HMAC-signed internal provisioning endpoint.
NODE_ENV—When production, the auth cookie is marked Secure (HTTPS only).

The session cookie is fixed: name authentication_token, httpOnly, sameSite=strict, maxAge 3 hours.

Data Model

The aggregates are described conceptually in Identity & Access; this section records their persistence-level detail.

Account

interface IAccount {
  name: string;        // unique
  email: string;       // unique
  password: string;    // bcrypt hash, cost factor 10
  memberships: IDomainMembership[]; // embedded subdocuments
  totpSecret?: string;
}

memberships is embedded, so a single findOne returns everything needed to mint a token.

Domain

interface IDomain {
  name: string;                     // unique tenant identifier (used everywhere by value)
  subdomains: ObjectId[];           // references to other Domain documents
  authStrategy: "internal" | "oidc";
  totpSecrets: IDomainTOTPSecrets;  // select:false — excluded from default queries
  ssoConfig?: ISSOConfig;           // clientSecret is select:false
  isVisibleFromOutside: boolean;    // default false
}
  • totpSecrets and ssoConfig.clientSecret carry select: false; they are returned only when a query explicitly re-selects them (.select("+totpSecrets"), .select("+ssoConfig.clientSecret")).
  • IDomainMembership is { domainName, role, externalId? } and is the value object embedded in Account.memberships.

Role

role.ts defines a ranked enumeration and the single comparison used for all authorization:

export const ROLE_WEIGHTS = {
  admin: 100, business_admin: 80, business_staff: 60, standard_customer: 10,
} as const;

export const hasRequiredRole = (held: Role, required: Role): boolean =>
  (ROLE_WEIGHTS[held] ?? 0) >= (ROLE_WEIGHTS[required] ?? Infinity);

Token Service

services/token.ts mints two token shapes, both signed with JWT_SECRET.

TokenClaimsLifetime
Standard (user)accountId, accountName, accountMemberships3 hours
DevicedomainName, buildingId, deviceName, deviceType, isBuildingSpecific1825 days

generateStandardToken re-reads the account before signing, so the embedded accountMemberships always reflect current state. Because memberships are baked into the token, a role change is not visible until the client obtains a fresh token — callers must re-authenticate (or re-fetch) after claiming a role.

Authentication Flows

Internal credentials

Registration hashes the password with bcrypt (genSalt(10)), rejecting duplicates with a 409 Conflict. Login compares with bcrypt.compare; a wrong password is a 400, an unknown account a 404. On success the controller signs a standard token and sets it as the httpOnly cookie.

OIDC / SSO

GET /auth/sso/login/:domainName is authentication-protected — SSO links a domain to an already-authenticated account rather than creating one.

The groups claim is mapped to a role in priority order:

IdP group(s)Role
admin, platform-adminadmin
business-admin, business_adminbusiness_admin
staff, business-staffbusiness_staff
anything else / nonestandard_customer

Known gap: SSO state is stateless base64

generateSSOLoginUrl encodes the PKCE verifier, username, and domain into the OAuth state as base64 JSON (marked TODO in code). This exposes the verifier to the browser and lets a crafted state be replayed. The intended design stores this server-side under a short-lived Redis key. See Domains.

TOTP role claiming

Roles above standard_customer are granted with a six-digit, time-based code derived from the domain’s totpSecrets (one secret per grantable role, generated at domain creation). resolveRoleFromOTP validates the submitted code against every domain’s secrets (window: 1); a match yields a { domainName, role } membership, which grantTOTPRoles applies by removing any existing membership for that domain and pushing the new one.

Domain Operations

Creation side effects

addDomain performs three steps:

  1. Generate totpSecrets for every role at or above business_staff.
  2. Persist the domain (ssoConfig is stored only for oidc strategy).
  3. Grant the creator a business_admin membership, and call the notification-service to initialise default preferences.

Subdomain resolution

getDomainSubdomains uses a MongoDB $graphLookup aggregation, so the hierarchy can be arbitrarily deep and is resolved in a single query.

Subscription rules

subscribe joins an account to a domain as standard_customer and notifies the notification-service. It rejects oidc domains — those grant membership only through the SSO callback. unsubscribe removes the membership and disables the corresponding notification preference.

Error Handling

models/error.ts defines a BaseError hierarchy carrying an HTTP code and a type: ValidationError (400), UnauthorizedError (401), ForbiddenError (403), NotFoundError (404), ConflictError (409), InternalError (500). Controllers and services throw these; the global error middleware serialises them into a consistent JSON envelope. See Auth Service: Middleware for the authorization and error-handling middleware in detail.
