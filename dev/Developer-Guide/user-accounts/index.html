Users and Account

Identity and Access Management (IAM) in CrowdVision is handled by Keycloak (credentials, MFA, federation) plus claims-gateway (token minting) and tenancy-service (domains and memberships). The legacy Mongo-backed auth-service that used to own all of this has been decommissioned. Because of our Database-per-Service architecture, it is still critical that identity and organizational permissions are structured so other services can securely authorize actions without cross-database queries — that contract now travels inside the RS256 JWT claims-gateway mints, not inside a shared database.

👤 The Account and Membership Model

Keycloak owns credentials and the account record itself; CrowdVision never stores a password. tenancy-service owns the relational projection of “who belongs to which domain, with what role” — a Postgres memberships table (account_id, domain_id, role, external_id), not an embedded array on an account document. A membership is written lazily: never at registration, but derived on first login (JIT provisioning) or via an invite-code redemption.

Why memberships still travel inside the JWT

Every login (or token refresh) needs to evaluate a user’s permissions without a database round-trip on the request hot path. claims-gateway resolves tenancy-service’s memberships once, at login, and bakes them into the signed token as a memberships:[{domain, role, externalId?}] claim — the same performance rationale the old design had (a single lookup gives the caller’s entire permission matrix), just moved out of Mongo and into the token-minting step.

  • The Trade-off: unchanged from the original design — write-complexity (a membership change needs a fresh token before it’s visible) traded for read-performance (zero-hop authorization on every other request).
  • Payload Ceiling: because memberships are embedded in the JWT, there’s still a practical ceiling on how many domains a single user can belong to before header size limits (~8KB) are breached. Same constraint, different transport.

⚖️ Role-Based Access Control (RBAC)

CrowdVision utilizes a Hierarchical Weight System to evaluate roles. Instead of explicitly defining what every single role can do, roles are assigned a numeric weight. If an action requires a business_staff role, the system checks if the caller’s role weight (in the relevant domain) is greater than or equal to the required weight. The weights are the single source of truth in auth-contracts/roles.json (Go) — every language that needs to evaluate a role (Node’s middlewares/authentication.ts normalization, agent-service’s app/roles.py) mirrors this same table.

Role NameInternal WeightPrivileges
Admin100Full system control, can manage other domains.
Business Admin80Organization owner, can manage buildings, sensors, and generate invite codes.
Business Staff60Facility managers, can edit room properties (capacity, temp thresholds).
Standard Customer10General users/viewers. By default, all new accounts are assigned this role.

🛡️ Authorization in Practice

Each Node service’s requireAuthentication verifies the JWT (RS256, against claims-gateway’s JWKS) and attaches the decoded, normalized claims to req.account — see Shared Auth Middleware for the verification pipeline. There is no separate requireAuthorization(level) middleware shared across services anymore; role checks are either inline (claims.CanIn(domain, role) in Go’s tenancy-service, using auth-contracts) or evaluated directly against req.account.accountMemberships where a route needs a domain-scoped check.

Contextual Evaluation (domain-scoped roles)

A user might be a business_admin in the “Main Campus” domain, but only a standard_customer in the “Engineering Lab” domain. The same token is valid for both — whichever service enforces the check filters the caller’s memberships down to the domain the request concerns, then compares weight.

⚠️ API Design Rule: URL Context Because authorization is evaluated from the token alone, a service cannot look up which domain a resource belongs to it wasn’t told. Therefore, protected routes that need a domain-scoped check take the domain as an explicit parameter rather than inferring it from another resource id.

  • DO: PATCH /domains/:domainName/rooms/:roomId
  • DON’T: PATCH /rooms/:roomId (nothing to check permissions against).

🔑 Role Elevation (Invite Codes)

When a user’s membership is first provisioned (JIT, on login) it defaults to whatever the domain’s join policy grants — typically standard_customer for an open-via-idp domain. To grant a higher role (or membership in an invite-only domain), tenancy-service mints single-use invite codes, replacing the old TOTP-based system entirely (see tenancy-service/CLAUDE.md’s note on why TOTP was dropped rather than ported):

  1. Invite Generation: A business_admin calls POST /domains/{domain}/invite-codes with the target role; tenancy-service mints a random code tied to that domain and role, with an expiry.
  2. Redemption: Any authenticated caller calls POST /invite-codes/{code}/redeem. The redemption is atomic at the database level (a single conditional UPDATE ... RETURNING, not a check-then-write in application code), so a code can only ever be consumed once even under concurrent attempts.
  3. Role comes from the code, never the redeemer — the same principle the old TOTP design had (the code encodes what the inviter authorized, not what the invitee claims).

The client’s QR-code UI still exists, but now encodes the raw invite code as its payload instead of a TOTP otpauth:// URI — a drop-in replacement requiring no changes to the QR-rendering component itself.

🛠️ Architecture Notes

  • Rate-limiting invite-code redemption is still worth doing — a malicious script could attempt many random codes against /invite-codes/{code}/redeem. Unlike the old TOTP design (a 6-digit code, ~1,000,000 combinations), invite codes are long random tokens, which makes brute-forcing impractical without rate-limiting being a hard requirement — but it’s still good defense-in-depth.
  • Per-domain BYO-IdP / OIDC federation and enterprise self-registration (the old auth-service’s /business/register, /auth/sso/*) were deliberately not carried forward — they have no tenancy-service equivalent today. If needed again, they need a fresh design against Keycloak’s Organizations/identity-brokering features, not a port of the old per-domain client-secret model.
  • Horizontal permissions (the hierarchical weight system assuming higher roles always inherit lower-role abilities) remains an open question, same as under the old design — evaluate if a future requirement needs role capabilities that don’t nest cleanly.
