Building Management Context

Owning service: twin-service · Subdomain type: Core · Persistence: MongoDB (twin-db)

The Building Management context models the physical reality being monitored: buildings, the rooms inside them, and the spatial geometry that places those rooms in the 3D digital twin. It is the structural backbone the rest of the platform decorates with live data.

Ubiquitous Language

TermMeaning in this context
BuildingA monitored physical structure, scoped to one or more domains.
RoomA bounded space inside a Building with a position.
Position / DimensionsThe geometry that renders a Room in the 3D twin.

The Building Aggregate

Building is the single aggregate root. A Room has no independent existence — it is created, mutated, and deleted only through its Building, which is the textbook reason to keep it inside the aggregate boundary rather than promoting it to its own collection.

Aggregate Root: Building

  • id is a service-generated UUID, decoupling the public identifier from MongoDB’s _id.
  • domains is an array of domain names (the shared identifier), making a building multi-tenant: it can be shared across organizations without duplication.
  • Invariant: every mutation to a Room must go through the Building, so the aggregate as a whole is always persisted in a consistent state.

Entity: Room

  • Identity is the id field, unique within its Building.
  • Holds only structural data — name, geometry. Note that live readings (temperature, occupancy) are not stored here; they belong to the Telemetry contexts. This separation keeps the twin model stable while readings churn continuously.

Value Objects: Coordinates and Dimensions

Both are immutable, identity-less geometry objects persisted as _id: false subdocuments. Two rooms with identical coordinates are not “the same position” in any meaningful sense — position is defined entirely by its values, which is exactly the value-object contract.

Multi-Tenancy as a Domain Rule

Because this context owns no copy of the identity model, it enforces tenancy purely from the JWT it receives and the domains array it stores. There is no call back to auth-service.

The Building.find({ domains: domainName }) query is indexed on domains; without that index every tenant lookup would be a full collection scan.

Relationships to Other Contexts

This context is an upstream supplier to three downstream contexts. The side effects below run synchronously inside addBuilding when a building is first created.

Pragmatic note

The cross-context seeding calls are fire-and-forget from the twin’s perspective: if a downstream call fails, the error is logged and building creation still succeeds. This favours availability over strict consistency, on the assumption that preferences and thresholds can be re-seeded later. It is a deliberate softening of the aggregate’s transactional boundary across the network.
