Identity & Access Context

Owning service: auth-service · Subdomain type: Supporting · Persistence: MongoDB (auth-db)

The Identity & Access context answers three questions: who is this user, which organizations do they belong to, and what may they do there. It is the upstream supplier of identity for the entire platform — it mints the JWT that every other context treats as its Published Language.

Ubiquitous Language

TermMeaning in this context
AccountA human user or system principal that can authenticate.
DomainAn organization, campus, or tenant. The unit of multi-tenancy.
MembershipThe link that places an Account inside a Domain with a specific Role.
RoleA ranked permission level used for authorization decisions.
Auth StrategyHow a Domain verifies identity: internal credentials or delegated oidc.

Aggregates

This context holds two aggregate roots — Account and Domain — that are intentionally kept separate. An Account references a Domain only by its name (the shared identifier), never by a hard database reference, so the two can be created, queried, and scaled independently.

Aggregate: Account

The consistency boundary around a single principal and its memberships.

  • memberships is embedded rather than referenced. A single read of the Account yields everything needed to mint a JWT, which keeps the authentication path to one query.
  • password and totpSecret are credential material; the latter is the per-account TOTP seed used for second-factor flows.
  • Invariant: an Account holds at most one membership per domainName. The SSO callback enforces this by replacing any prior membership for the domain rather than appending a second one.

Aggregate: Domain

The consistency boundary around an organization and the rules under which members join it.

  • authStrategy is the central policy switch. It determines the entire sign-in flow and which value objects are populated.
  • totpSecrets and ssoConfig.clientSecret are marked select: false in the schema — they are never returned by default queries and must be explicitly re-selected. This keeps secret material from leaking through ordinary reads.
  • subdomains references other Domain documents by ObjectId; hierarchy resolution uses MongoDB $graphLookup, so the tree may be arbitrarily deep.
  • Invariant: an internal domain never carries an ssoConfig; an oidc domain blocks manual subscription. These rules live in the Service layer, not on the schema.

The Role Value Object

Role is a value object backed by a weighted ordering rather than a free string. Authorization is a comparison of weights, which makes “at least this powerful” the natural primitive.

export const ROLE_WEIGHTS = {
    admin: 100,
    business_admin: 80,
    business_staff: 60,
    standard_customer: 10,
} as const;

export const hasRequiredRole = (userRole: Role, requiredRole: Role): boolean =>
    (ROLE_WEIGHTS[userRole] ?? 0) >= (ROLE_WEIGHTS[requiredRole] ?? Infinity);

hasRequiredRole is the single domain rule for permission checks; the authorization middleware evaluates it against the memberships carried in the verified token.

Domain Services

Published Language: the JWT

The output of this context is a signed token whose payload is the contract every other context conforms to. Treat its shape as a public interface — changing it ripples across the whole platform.
