Strategic Design

CrowdVision is modelled with a Domain-Driven Design (DDD) mindset. The system is not a single monolithic model; it is a set of collaborating models, each owned by a service and each speaking its own dialect of a shared Ubiquitous Language. This page describes the strategic layer — the subdomains, the bounded contexts, and the relationships between them. The tactical layer (aggregates, entities, value objects, and domain services) is documented per context on the following pages.

A pragmatic stance

DDD is used here as a design compass, not a rulebook. The strategic boundaries below are firm and have proven stable. The tactical patterns are applied where they earn their keep: aggregates and value objects are expressed as Mongoose subdocuments and typed structs rather than a formal domain layer, repositories are realised implicitly through the Service and Model layers, and domain events are carried over Redis pub/sub messages. Some boundaries were intentionally left soft because the product ideas were still maturing. Where the implementation diverges from textbook DDD, the per-context pages call it out explicitly.

Domain and Subdomains

The overall domain is real-time monitoring of physical spaces through a digital twin. We classify the subdomains by how much competitive value they carry, which in turn drives how much modelling effort each receives.

SubdomainTypeOwning ServiceRationale
Spatial Digital TwinCoretwin-serviceThe 3D building model is the product’s differentiator and receives the richest model.
Telemetry Ingestion & ThresholdsCoresensor-serviceTurning raw sensor signals into meaningful, bounded readings is central business logic.
Telemetry DistributionCorecontracts-servicePer-building stream shaping is what makes the live dashboard feel tailored.
Identity & AccessSupportingauth-serviceMulti-tenant access control is essential but not a differentiator.
AlertingSupportingnotification-servicePush alerting supports the core experience without defining it.
Knowledge & AssistanceSupportingagent-serviceThe LLM assistant augments the product; it is explicitly experimental.
Real-Time TransportGenericsocket-serviceA commodity WebSocket relay with no domain model of its own.

Bounded Contexts

Each core and supporting subdomain is realised as exactly one bounded context, and each bounded context maps one-to-one onto a deployable service with a private database. This alignment is deliberate: a context boundary is also a network and persistence boundary, so a model can never silently leak into another context through a shared table.

The Real-Time Transport context (socket-service) is omitted from the context map: it carries messages between contexts but holds no model and enforces no invariants.

Context Map

The relationships between contexts are described with the standard DDD integration patterns. They tell a new contributor who depends on whom and which side is allowed to dictate the contract.

Upstream → DownstreamPatternIntegration
Identity → all other contextsPublished LanguageThe signed JWT payload (memberships, roles, domains) is the published language every context conforms to. Downstream contexts are Conformists: they verify and read the token but never reshape it.
Building Management → Telemetry IngestionCustomer / SupplierOn building creation the twin pushes a threshold clone (PUT /thresholds/buildings/:id). The twin is the customer driving the contract.
Building Management → Telemetry DistributionCustomer / SupplierThe twin seeds default column preferences (POST /preferences/init/:id) and exposes its own structural metrics (roomName, roomMaxOccupancy) as part of the metric catalog.
Telemetry Ingestion → Telemetry DistributionOpen Host Service + Published LanguageThe sensor service publishes a self-describing Metric Contract at GET /contracts. The contracts service discovers and aggregates these, so new metric types appear without code changes downstream.
Building Management → AlertingCustomer / Supplier (Anti-Corruption)Alerting resolves a building’s domains by calling the twin (GET /twin/domain/:buildingId) and translates the response into its own subscription concepts.
Building Management → Knowledge & AssistanceAnti-Corruption LayerThe agent reaches live building data only through read-only tools that wrap the twin API, keeping the twin model out of the agent’s domain.

The Shared Identifier Convention

Across the entire platform, a Domain is referenced by its name string, never by a database identifier. This is the closest thing CrowdVision has to a Shared Kernel, but it is deliberately minimal: a single, stable, human-readable string rather than a shared code library.

Because the domain name travels inside the JWT and is stored verbatim in every context’s database, multi-tenant isolation is enforced symmetrically and without any cross-database joins or synchronous service-to-service calls on the hot path. Each context answers “may this caller see this data?” entirely from data it already owns.

Tactical Patterns at a Glance

The building blocks below recur throughout the per-context pages. Their CrowdVision implementations are listed so the mapping from theory to code is unambiguous.

DDD Building BlockHow it is realised in CrowdVision
Aggregate RootA top-level Mongoose model (or Rust struct) that owns a consistency boundary, e.g. Account, Building, BuildingThreshold.
EntityAn object with identity that lives inside an aggregate, e.g. Room within Building.
Value ObjectAn immutable, identity-less subdocument, e.g. Coordinates, Dimensions, SubscriptionPreference.
Domain ServiceStateless logic that does not belong to a single entity, e.g. tokenService, totpService, the sensor SensorKernel.
RepositoryNot a dedicated class; the Service + Model layers together play the repository role (see The Microservices.
Domain EventA message published to Redis, e.g. telemetry:raw, telemetry:filtered:{building}, notifications.

Where the model is intentionally anemic

Several aggregates are persistence-shaped rather than behaviour-rich — the schema carries the data and the Service layer carries the rules. This was a conscious trade-off for development speed while the bounded contexts were still settling. The pages that follow flag the spots where behaviour should eventually migrate back onto the aggregate.
