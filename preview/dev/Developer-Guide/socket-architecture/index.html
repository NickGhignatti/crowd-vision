Socket Service Architecture

Bounded context: real-time transport (generic) · Stack: Node.js / Socket.IO · Code-level walkthrough: Socket Service

The socket-service is the bridge between the broker and the browser. It carries per-building telemetry and alerts to connected clients over WebSockets. It is classified as a generic subdomain (see Telemetry Distribution: pure transport, with no domain model and no database.

Architectural Style

Deliberately the simplest service in the system: no persistence and no domain model. Everything the service does splits cleanly into two kinds of code, and index.ts says so in its own header comment: “Composition root (imperative shell): build dependencies, wire the pure handlers to them, and start listening. No logic lives here — that’s in core/ and handlers/, which is why this file is excluded from coverage.”

  • The functional core (core/relay.ts) is three pure functions — roomForBuilding, roomForDomain, buildingIdFromChannel — string manipulation only, no I/O, no dependencies.
  • The imperative shell is everything that touches the outside world: auth.ts decodes the mesh-injected x-gateway-claims header on the WebSocket handshake (Istio’s RequestAuthentication already verified it once, upstream — the same trust model every other migrated service uses); handlers/connection.ts, handlers/telemetry.ts, and handlers/notification.ts do the actual work of joining rooms and emitting to sockets.
  • index.ts is the composition root: it registers the auth check as Socket.IO handshake middleware, wires each Redis channel to its handler, and starts listening — nothing else.

Key Architectural Decisions

  • Pure transport, no model. Treating real-time delivery as a generic subdomain keeps all domain logic out of it and concentrates modelling effort on the core services.
  • Identity trusted from the mesh, not re-verified. The handshake decodes an already-verified claims header instead of checking a JWT signature itself; the resulting identity is server-held for the life of the connection, so a client can never join a domain or building room it isn’t entitled to.
  • Room-scoped telemetry, domain-scoped notifications. Clients receive readings only for buildings they have joined, and notifications only for their own domain; tenants stay isolated. Only a notification with no domain (a system-wide message) is broadcast.
  • Stateless relay. The service holds no durable state, so an instance can be replaced freely.
  • Scaling is an explicit concern. Running more than one instance requires sticky sessions for the handshake and a broker-backed adapter to deliver across instances — a known extension point rather than a current capability.

Integration

  • Subscribes to the alert channel and the per-building telemetry feeds (telemetry:filtered:<id>), and emits them to clients — see Communication & Data Flow.
  • Holds no database and calls no other service synchronously.

For the channel-to-event mapping, the client subscription protocol, and scaling notes, see the Socket Service internals page.
